# VKE 실전형 Scale-out BMT 결과

## 시나리오

하나의 서비스 URL인 `http://web-bmt.default.svc.cluster.local/` 뒤에 nginx 웹 Pod를 붙였다.

처음에는 backend 웹 Pod 50개로 운영했고, 별도 부하 발생용 Pod 200개가 같은 URL로 동시에 접속했다. 이후 웹 Pod를 200개로 4배 확장한 뒤 동일한 부하를 다시 걸어 전후 수치를 비교했다.

## 전후 비교

| 구분 | Web Pod | Loadgen Pod | 총 요청 수 | 성공률 | 오류 수 | 평균 응답 | p95 응답 |
|---|---:|---:|---:|---:|---:|---:|---:|
| 확장 전 | 50 | 200 | 4,305 | 85.8% | 610 | 569.0ms | 3000.0ms |
| 확장 후 | 200 | 200 | 4,175 | 91.6% | 350 | 368.6ms | 2473.1ms |

여기서 **총 요청 수**는 초당 요청 수(RPS)가 아니라, 해당 부하 테스트 1회 동안 부하 발생 Pod들이 보낸 전체 요청 시도 수다.

예를 들면 확장 전에는 전체 4,305번 요청 중 3,695번이 HTTP 200으로 성공했고, 610번은 실패 또는 타임아웃으로 집계됐다. 그래서 성공률이 85.8%다.

## 해석

- 같은 Service URL과 같은 200개 부하 발생 Pod 구조를 유지했다.
- backend 웹 Pod만 50개에서 200개로 4배 확장했다.
- 확장 후 성공률은 85.8%에서 91.6%로 상승했다.
- 오류 수는 610건에서 350건으로 감소했다.
- 평균 응답 시간은 569.0ms에서 368.6ms로 낮아졌다.
- p95 응답 시간은 3000.0ms에서 2473.1ms로 낮아졌다.

## 정리 상태

BMT 완료 후 VKE 클러스터와 연결된 worker 리소스를 삭제했다. 최종 확인 결과 실험용 VKE 클러스터와 worker는 남아 있지 않았고, 기존 `ntopng-base-image` 서버만 남아 있었다.
